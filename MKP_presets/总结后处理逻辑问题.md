### 2.3 ironingMicroLiftEnabled 字段提交时的数据类型

**关键发现 - 存在数据类型不一致问题**:

1. **后端 Go**: `WipingConfig.IroningMicroLiftEnabled` 是 `bool` 类型
2. **TOML 文件**: 存储为 `true`/`false`（由 `fmt.Sprintf("%t", ...)` 生成）
3. **前端读取**: `tomlToConfig` 函数（tomlConfigUtils.ts 第148-180行）对 `IroningMicroLiftEnabled` 没有特殊转换，直接使用从 toml 读取的 bool 值
4. **前端 UI 渲染**: 使用 `segmented` 控件，选项是字符串 `"off"`/`"on"`
5. **前端提交**: 当用户点击"开启"时，`config['IroningMicroLiftEnabled']` 变为字符串 `"on"`；点击"关闭"时变为字符串 `"off"`
6. **前端保存转换**: `configToToml` 函数（tomlConfigUtils.ts 第187-206行）对 `IroningMicroLiftEnabled` **没有特殊转换**（不像 `UseWipingTowers` 和 `FastTowerMode` 有专门的 bool→string 转换）

**对比其他字段的特殊处理**（tomlConfigUtils.ts 第197-201行）:
```typescript
// 这两个字段有特殊转换，但 IroningMicroLiftEnabled 没有！
if (configKey === 'UseWipingTowers') {
  value = typeof value === 'boolean' ? (value ? 'tower' : 'disk') : value;
} else if (configKey === 'FastTowerMode') {
  value = typeof value === 'boolean' ? (value ? 'fast' : 'follow') : value;
}
```

**结论**: 当用户通过 UI 修改 `IroningMicroLiftEnabled` 后，提交给后端的值是字符串 `"on"` 或 `"off"`，而后端期望的是 `bool` 类型。这可能导致 TOML 反序列化失败或值丢失。

## 二、虚拟参数为什么保存不了？

### 根本结论

虚拟参数的保存机制是**设计如此**：虚拟参数本身没有 TOML 字段（无 `configKey`/`tomlKey`/`jsonKey`），因此不会被直接写入 TOML。其值通过 `onChangeMulti` 转换为一个或多个真实参数的值，由真实参数负责写入 TOML。

但需区分三种情况：
| 虚拟参数 | 是否有 configKey | 自身是否写入 TOML | 间接写入的真实参数 |
|---------|-----------------|------------------|-------------------|
| `outerStructure` | **无** | 否（被跳过） | RibOuterWall + SheathMode |
| `wipingMode` | **有** (UseWipingTowers) | 是 | + UseDiskWipeFlag |

export const replaceRetractInGcode = (gcode: string, newValue: number): string => {
  return gcode.replace(/(G1\s+E)-?\d+\.?\d*/, `$1${newValue}`);
};

export const replaceExtrudeInGcode = (gcode: string, newValue: number): string => {
  const lines = gcode.split('\n');
  let replaced = false;
  const newLines = lines.map(line => {
    if (!replaced) {
      const match = line.match(/(G1\s+E)(\d+\.?\d*)/);
      if (match && parseFloat(match[2]) > 0) {
        replaced = true;
        return line.replace(/G1\s+E\d+\.?\d*/, `G1 E${newValue}`);

    }
    return line;
  });

**可以移除，且推荐移除**。理由：

1. **`transform: "mkpRetractGcode"` 字段是死元数据**：前端没有任何代码检查 `param.transform === 'mkpRetractGcode'`，实际联动通过 `useTomlManager.ts` 中硬编码的 `if (key === 'MKPRetract')` 实现。

3. **当前设计的问题**：前端在用户修改 MKPRetract 时立即重写 `CustomMountGcode`/`CustomUnmountGcode` 两个 G-code 字符串字段。这导致：
